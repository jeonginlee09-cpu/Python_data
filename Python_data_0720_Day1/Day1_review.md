# Day1 평가

- 이름: 이정인
- 과제 범위: Practice 1 ~ Practice 4

## Practice 1

리스트 방식과 제너레이터 방식의 차이를 알아보기 위해 동일한 데이터를 처리하는 두 가지 코드를 작성했습니다. `practice1.py`에는 제너레이터 방식을, `practice1_com.py`에는 리스트 방식을 적용했습니다.

처음 두 방식을 비교했을 때는 최대 메모리 사용량에 큰 차이가 있었습니다. 그러나 각 파일을 완성한 뒤 다시 실행했을 때는 제너레이터 방식의 최대 메모리 사용량이 약 20MB까지 증가했습니다. 이러한 변화가 발생한 원인을 정확히 찾지 못한 점이 아쉬웠습니다. 앞으로 측정 범위와 실행 환경, 집계 과정에서 생성되는 객체 등을 확인하여 원인을 더 탐구하고 싶습니다.

## Practice 2

먼저 원본 데이터에서 오염된 값을 눈으로 확인한 뒤, Pydantic을 활용하여 데이터 검증 코드를 작성했습니다. 유효한 데이터와 오염된 데이터를 분리하고, 각 데이터가 어떤 검증 규칙을 통과하지 못했는지 확인할 수 있었습니다.

## Practice 3

`PERMANENT_FAILURE_IDS`를 활용하여 임의로 지정한 ID 17과 43이 항상 실패하도록 설정했습니다. 실행 결과 60건 중 2건이 최종 실패한 것을 확인했으며, 실패한 두 건이 `dead_letter.json`에 정상적으로 저장된 것도 확인했습니다. 이를 통해 재시도 후에도 실패한 데이터를 별도로 격리하는 방법을 익힐 수 있었습니다.

## Total1

아직 복습이 필요한 부분이지만, 제시된 단계에 따라 비동기 ETL 파이프라인을 작성하고 성공 판정 기준에 맞춰 실행, 테스트, Ruff 검사까지 완료했습니다.

한 가지 아쉬운 점은 Practice 3의 영구 실패 데이터를 `PERMANENT_FAILURE_IDS`로 처리하여 Extract 단계에서 제외한 것입니다. 이 데이터는 Transform 단계에 도달하지 않기 때문에 Pydantic 검증 결과의 `invalid` 건수에는 포함되지 않습니다. 앞으로는 수집 실패와 검증 실패의 차이를 더 명확히 이해하고, Pydantic 검증에 실패하는 오염 데이터 2건을 별도로 만들어 `invalid=2`로 처리하는 방법도 탐구하고 싶습니다.
